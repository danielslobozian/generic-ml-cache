Scopes and sessions are future observability and organization features.

They should not affect cassette identity.

## Scopes

A scope is a cache/reporting namespace selected by a scope token.

Scope tokens are:

- generated by gmlcache,
- treated as secrets by callers,
- stored only as hashes,
- not user accounts,
- not authentication,
- not authorization.

A scope can own sessions and cached data. A public scope can exist when no token
is provided.

## Sessions

A session is a workflow boundary inside a scope.

Sessions are generated by gmlcache and require a scope token. They group multiple
executions into a single reportable unit.

Examples of session boundaries:

- one CI pipeline run,
- one workflow engine run,
- one evaluation batch,
- one feature-generation pass,
- one scripted experiment.

## Hierarchy

```text
Scope
  owns Sessions
  owns or references Cassettes

Session
  observes Executions

Execution
  records or reuses a Cassette
```

## Reports

A session report can show:

- executions,
- cache hits,
- records,
- non-cacheable calls,
- adapter/model breakdown,
- token usage,
- client-reported cost,
- unknown-cost executions.

A scope report can aggregate many sessions.

## Invalidation

Invalidating a scope token removes that scope’s cached data and metadata. It is a
storage operation, not an authentication operation.
