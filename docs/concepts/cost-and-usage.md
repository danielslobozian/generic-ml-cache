Cost and usage data are observability features.

They help callers understand what detached ML executions consumed and what replay
saved. They do not decide cache correctness.

## Current usage envelope

When an adapter exposes structured usage, gmlcache stores a normalized envelope in
the cassette. The current envelope can include:

- input tokens,
- output tokens,
- cache-read tokens,
- cache-write tokens,
- reasoning tokens,
- client-reported cost,
- the raw adapter usage block.

Unknown values stay unknown. They are not converted to zero.

## Token categories

### Input tokens

Tokens submitted to the model for the execution.

### Output tokens

Tokens generated by the model.

### Cache-read tokens

Provider-side cache-read tokens reported by the adapter when available. These are
not gmlcache replay hits. They describe the underlying provider/client’s own
prompt-cache behavior.

### Cache-write tokens

Provider-side cache-write tokens reported by the adapter when available.

### Reasoning tokens

Additional reasoning-token counts when the adapter exposes them. Not all adapters
report this value.

## Cost categories

### Client-reported cost

A cost value reported by the client or provider. This is advisory and should be
shown as an estimate, never as authoritative billing.

### Estimated saved cost

When a cassette with recorded usage is replayed, gmlcache can estimate what was
saved by not calling the client again. This estimate depends on recorded client
usage and recorded client cost. Replays without usage metadata are excluded from
that estimate.

### Unknown cost

If an adapter does not report cost, the cost is unknown. gmlcache should not
invent provider pricing.

## Sanitized example

A cassette may contain a normalized usage block like:

```json
{
  "input_tokens": 2538,
  "output_tokens": 2395,
  "cache_read_tokens": 54445,
  "cache_write_tokens": 11561,
  "reasoning_tokens": null,
  "cost_usd": 0.2160155,
  "raw": { "...": "adapter-specific usage" }
}
```

A stats report may then summarize saved usage from replayed cassettes:

```text
saved: from 7 replay(s) — input 111777, output 24078, cache-read 480730 tokens
       ~ $0.9377 (from client cost estimates; not authoritative)
       (6 replay(s) had no recorded usage and are not counted)
```

These numbers are illustrative. Documentation should not include machine paths,
real cache keys, or personal prompts.

## Future session and scope reports

Future reports can aggregate the same usage data by session and scope:

```text
Session report
  executions: 12
  hits: 7
  records: 5
  input tokens: ...
  output tokens: ...
  cache-read tokens: ...
  client-reported cost: ...
  unknown-cost executions: ...
```

The aggregation is observational. It must not affect replay decisions.
